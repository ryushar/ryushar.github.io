Rendering particles in ryuten.io

April 12, 2024 (Today)

There are many types of elements in Ryuten, one of them being particles. Depending upon the size and zoom level of a player, a lot of these can become visible at once in the game.

Ryuten uses PIXI.js library for rendering the game visuals. Pixi.js uses WebGL under the hood for rendering stuff on the canvas element and provides an easy to understand high level API to help speed up the development process. It also has a low level API for the cases where you need more control and performance.

A very simple expanation of how WebGL renders stuff

  • You provide some vertices
  • These vertices are processed and then used to create primitives
  • These primitives are then rasterised into fragments
  • Each pixel in the fragment is then processed to output the final pixel data

Rendering particles using PIXI.JS normally

  • PIXI.JS uses rectangles (i.e, 2 triangles, 4 vertices) to create a primitive and overlays it with a texture.

  • Let us assume that we are rendering 50px x 50px particle.

  • Pixel count = 50 * 50 = 2500 pixels in total

  • So, the vertex shader deals with 4 vertices and the fragment shader deals with 2.5k pixels.

Rendering with PIXI.JS normally vs using the low level API

  • Since we have more control using the low level API, instead of a rectangle we are going to use a hexagon.

  • Let us assume that we are rendering 50px x 50px particle.

  • Pixel count = ((3 x √3) / 2) x (25 x 25) = ~1624 pixels in total

  • This time, the vertex shader has to deal with 6 vertices and the fragment shader deals with 1.6k pixels.

Note: No texture clipping takes place while rendering as there is always extra headroom available due to some reasons related to mipmapping.

Comparision

  • While the vertices increased by a small count of 2, the pixels decreased from 2.5k to 1.6k (roughy 36% decrement)
  • Since there didn’t seem to be an accurate way available to measure the GPU stats, I decided to disable v-sync and remove FPS limit from the browser and let it render as many frames per second as possible using both the methods.
  • With the normal method, the game was able to get ~160 FPS on an average
  • While with the low level API method, the game was able to get ~200 FPS on an average.
  • Both the scenes are exactly the same with roughly 2.5k particles on the screen.

Conclusion

Using the lower level API surely does require extra brain power, time and increases maintenance cost, but in return in provides you with power and control. In this case, we were able to improve our particle rendering system by roughly 25%.
